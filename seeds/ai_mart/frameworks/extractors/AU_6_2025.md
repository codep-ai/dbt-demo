## 2. Fetch configuration (TinyFish)

```yaml
fetch:
  primary_url: https://www.industry.gov.au/publications/guidance-for-ai-adoption/guidance-ai-adoption-implementation-practices
  fallback_urls:
    - https://www.industry.gov.au/sites/default/files/2025-10/guidance-for-ai-adoption-implementation-practices.pdf
  content_type: auto  # HTML preferred; fall back to PDF if HTML unavailable or thin
  wait_for_selector: main  # page renders practices in <main>
  javascript: enabled
  timeout_seconds: 60
  retry_on_failure: 2
  capture:
    - full_page_text
    - content_sha256  # required for change detection
    - retrieved_at_utc
```

If `primary_url` returns HTTP 4xx/5xx or content length < 5 KB, fall back to `fallback_urls[0]` (the PDF). If both fail, **abort the run** and open a high-priority ticket — do not produce a candidate diff from thin or missing content.

## 3. AI extraction prompt (runs on TinyFish output)

> System prompt for the extractor AI (Claude Sonnet or Gemini 2.5 Pro). Framework-specific prompt; do not reuse across frameworks without review.

```
You are extracting AI governance control data from an Australian Government
publication. The source is the "Guidance for AI Adoption — 6 Essential
Practices" document, published October 2025 by the Department of Industry,
Science and Resources.

Your task: read the provided page content and emit exactly 6 rows — one per
Essential Practice — in the schema below. If you find fewer than 6 practices
or more than 6, emit what you find and flag the count discrepancy in the
`notes` column of every row you emit.

=== OUTPUT SCHEMA (CSV, 18 columns, header row first) ===

framework_code,framework_name,framework_publisher,jurisdiction_code,country_code,effective_from,is_mandatory,source_url,control_id,control_name,control_description,control_category,obligation_family,mandatory_records,source_section,retrieved_date,status,notes

=== CONSTANT VALUES (pre-populate on every row) ===

framework_code: AU_6_2025
framework_name: Guidance for AI Adoption — 6 Essential Practices
framework_publisher: Australian Government Department of Industry Science and Resources
jurisdiction_code: AU
country_code: AU
effective_from: 2025-10-08
is_mandatory: false
source_url: https://www.industry.gov.au/publications/guidance-for-ai-adoption/guidance-ai-adoption-implementation-practices
retrieved_date: <today's date in ISO 8601>
status: complete  (unless you flag ambiguity — then 'top_level_only' or 'stub')

=== PER-ROW EXTRACTION RULES ===

For each of the 6 Essential Practices, extract:

1. control_id:
   - Format: "P1", "P2", "P3", "P4", "P5", "P6"
   - Stable — these IDs are our internal identifiers and MUST match this
     numbering even if the publisher renames or reorders practices. If the
     publisher's ordering differs from P1-P6, map by semantic meaning, not
     by position, and note the discrepancy.

2. control_name:
   - The short practice title, verbatim from the publisher's heading.
   - Example: "Decide who is accountable"
   - Do NOT paraphrase or summarise.

3. control_description:
   - One-sentence description of what the practice asks organisations to do.
   - Prefer verbatim from the publisher; if the publisher gives only a long
     multi-paragraph section, produce a single-sentence faithful summary
     that does not add interpretation.
   - Must end in a full stop.

4. control_category:
   - Single-word category reflecting the practice's nature:
     P1 → "Accountability"
     P2 → "Impact"
     P3 → "Risk management"
     P4 → "Transparency"
     P5 → "Testing & monitoring"
     P6 → "Human oversight"
   - These map to the publisher's own section framing. If the publisher's
     section framing changes (new subheadings), adjust to match; flag in notes.

5. obligation_family:
   - Map to exactly one value from this closed set:
       accountability
       impact_assessment
       risk_management
       transparency
       testing_monitoring
       human_oversight
       redress
       third_party_supply_chain
       data_governance
       recordkeeping
   - Fixed mapping for AU_6_2025:
       P1 → accountability
       P2 → impact_assessment
       P3 → risk_management
       P4 → transparency
       P5 → testing_monitoring
       P6 → human_oversight
   - Do NOT deviate from this mapping without human review. If the publisher
     restructures the practices such that this mapping becomes wrong, flag
     via notes and emit your best mapping; the PR reviewer decides.

6. mandatory_records:
   - Pipe-separated list of artefacts the publisher says organisations
     should produce to evidence the practice.
   - Extract every distinct record type mentioned in the practice's
     implementation section.
   - Examples from the current published content:
       P1: "AI governance policy|job descriptions|executive accountable official assignment|responsible-parties register|governance forum minutes"
       P2: "AI screening/intake form|impact assessment record|stakeholder-identification record"
   - Do NOT invent records the publisher doesn't mention.

7. source_section:
   - Pointer into the source document.
   - Format: "Practice N — <short label>"
   - Example: "Practice 1 — Accountability"

8. notes:
   - Empty for a clean extraction.
   - Populate with short sentences flagging:
       • Any discrepancy between current extraction and the existing seed row
         (comparison input provided below)
       • Any version/structure change detected (new practice count, renumbering, etc.)
       • Paraphrasing you had to do because the publisher's text didn't fit
         one sentence

=== EXISTING SEED ROWS (REFERENCE FOR DIFFING) ===

[Extractor runtime injects the current 6 AU_6_2025 rows from the seed here,
 verbatim. The AI compares its extraction vs these rows and populates
 `notes` with any meaningful differences.]

=== DIFF OUTPUT REQUIREMENT ===

After emitting the 6 CSV rows, emit a separate JSON block summarising the diff:

{
  "framework_code": "AU_6_2025",
  "retrieved_date": "<today>",
  "source_content_sha256": "<from TinyFish capture>",
  "rows_extracted": 6,
  "rows_matching_seed": <count>,
  "rows_with_changes": <count>,
  "changes": [
    {
      "control_id": "P2",
      "field": "control_description",
      "before": "...",
      "after": "...",
      "confidence": "high|medium|low",
      "reviewer_action_suggested": "accept|reject|verify_against_source"
    }
  ],
  "structural_flags": []  // e.g. "new_practice_added", "practice_renumbered"
}

If `rows_matching_seed == 6` and `structural_flags` is empty, the pipeline
short-circuits: no PR is opened, only the hash and retrieved_date are
updated in sys_common_config. If any change is detected, a PR is opened
with the full CSV + JSON diff for human review.

=== INVARIANTS ===

- Never auto-merge. You are producing a proposal for human review, not a
  committed change.
- Never mutate published rows if the framework version has changed. Flag
  structurally and let the reviewer create a new framework_code.
- Preserve the citation chain. Every row must link back to source_section.
- If your confidence on any field is "low", set status to "stub" on that
  row and populate notes with the ambiguity.
- If the source URL returns 404 or has been permanently redirected, abort
  the extraction and flag via structural_flags.
```

## 5. Known edge cases for AU_6_2025

1. **Document hosted as both HTML and PDF.** HTML rendering is primary; PDF is the fallback. PDF extraction has historically been noisier — apply `status: stub` on any row where PDF-extracted text is ambiguous.
2. **Publisher may add sector-specific supplements** (e.g. "Guidance for financial services" as a companion document). These are **out of scope for AU_6_2025** — if discovered, flag via `structural_flags` as `"sector_supplement_detected"` and leave for the reviewer to decide whether to author a new framework entry.
3. **The 6 practices map 1:1 to the legacy VAISS 10 guardrails (AU_VAISS_10_2024)** but with different framing. Do not conflate — `AU_6_2025` and `AU_VAISS_10_2024` coexist as separate `framework_code` values. Reviewer owns the mapping between them if needed.
4. **Version marker on the page:** currently labelled "Version 1.0, October 2025." If the extractor sees "Version 2.0" or a different month/year, treat as a version change (see §1 invariant).
5. **Reading level / plain language variants:** the publisher sometimes ships simplified versions for SMEs. The canonical source for the Implementation Practices document is always the URL in §2. If redirected to a simpler "Foundations" page, that is a different framework — abort.

## 6. Run-record schema

Every TinyFish run of this extractor produces a record for audit:

```json
{
  "run_id": "<TinyFish run_id>",
  "framework_code": "AU_6_2025",
  "dag_run_id": "<Airflow dag_run_id>",
  "started_at_utc": "...",
  "completed_at_utc": "...",
  "source_url": "https://...",
  "fetched_url": "https://...",  // may differ if fallback used
  "content_sha256": "...",
  "previous_content_sha256": "...",
  "extraction_model": "claude-sonnet-X | gemini-2.5-pro",
  "extraction_prompt_sha256": "<sha256 of this file>",
  "rows_extracted": 6,
  "rows_with_changes": 0,
  "pr_url": "https://github.com/<org>/dbt-demo/pull/NNN",  // null if no changes
  "reviewer_verdict": "pending | approved | rejected",
  "reviewer_verdict_at_utc": null
}
```

Records land in the trace ledger (`traceability/ledger.py`) under event type `governance.framework_refresh` and are archived to the S3 bronze (Object Lock COMPLIANCE) per the immutability rules. Run records are the audit evidence that proves "we checked the regulator site on Date X, found hash Y, and either merged or chose not to merge the changes."
